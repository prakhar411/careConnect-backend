1. Project Overview

CareConnect is a comprehensive healthcare staffing platform that connects three types of users — Patients, Nurses, and Organizations (hospitals, clinics, nursing homes). The platform facilitates nurse hiring, appointment booking, credential verification, compliance tracking, and payment management.

User Role Description Key Activities
Patient Individuals requiring home healthcare or medical assistance Book nurses, manage appointments, upload medical records
Nurse Certified healthcare professionals seeking employment Browse jobs, apply, manage credentials, receive payments
Organization Hospitals, clinics, nursing homes, care centers Post jobs, review applications, manage compliance and credentialing

3. Database Design Goals

  1. Role Separation — A single users table handles authentication for all three roles. Each role has a dedicated profile table linked via a 1:1 relationship. This avoids null columns and keeps the auth table clean.
  2. Data Integrity — Foreign key constraints with appropriate CASCADE and SET NULL rules ensure referential integrity across all 11 tables.
  3. No Redundancy (3NF) — Common fields (email, password, role) are stored only in users. Role-specific fields live in their respective profile tables.
  4. Scalability — Strategic indexes on frequently queried columns (status, foreign keys, expiry dates) ensure query performance as data grows.
  5. Audit Trail — All major tables include created_at and where applicable updated_at timestamps for auditability and compliance.
  6. Financial Accuracy — All monetary columns use DECIMAL(10,2) instead of FLOAT to avoid floating-point precision errors.

10. Design Decisions & Rationale

10.1   Single users Table for All Roles

Rather than creating three separate user tables (patients, nurses, org_admins), a single users table with a role ENUM column is used. Role-specific data is stored in dedicated profile tables. This approach:

  • Simplifies authentication — one table, one email uniqueness constraint
  • Keeps the JWT token strategy consistent across all roles
  • Avoids the risk of the same email appearing in multiple tables
  • Makes login routing clean: check role, redirect to appropriate dashboard

10.2   Nullable nurse_id in appointments

The nurse_id foreign key in appointments is nullable. A patient can submit an appointment request specifying care needs, and a nurse is matched and assigned later by the organization. This design supports two workflows:

  • Direct booking — patient selects a specific nurse at booking time
  • Demand-based matching — patient describes needs; platform assigns the best-fit nurse

10.3   medical_records.uploaded_by References users

The uploader can be a patient uploading their own documents, or a nurse uploading clinical documentation during a visit. By pointing to users directly (rather than nurse_profiles), both scenarios are covered without needing two separate foreign key columns.

10.4   Composite Unique Constraint on nurse_applications

The composite unique key (nurse_id, job_id) ensures a nurse cannot apply to the same job more than once. This constraint is enforced at the database level, not just in application code, which prevents race conditions in concurrent requests.

10.5   CASCADE vs SET NULL on Delete

  • ON DELETE CASCADE is used where child records have no meaning without the parent — e.g., deleting a nurse deletes their applications and credentials automatically.
  • ON DELETE SET NULL is used where child records retain historical value even after the parent is removed — e.g., a payment record for a nurse who left the platform should still exist for financial audit purposes.

10.6   DECIMAL for Monetary Values

All financial columns (amount, salary_min, salary_max, tax_deduction) use DECIMAL(10,2) instead of FLOAT or DOUBLE. FLOAT and DOUBLE use binary floating-point representation which can introduce rounding errors in financial calculations (e.g., 0.1 + 0.2 ≠ 0.3 exactly). DECIMAL stores values as exact fixed-point numbers.

10.7   Hibernate DDL Strategy

The application uses spring.jpa.hibernate.ddl-auto=update which allows Hibernate to automatically create or alter tables based on entity changes without dropping existing data. For production deployment, this should be changed to validate and schema migrations should be managed through the provided DDL script.
